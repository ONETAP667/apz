# Лабораторна робота 3 — аналіз CQS

## Що змінилося в структурі проєкту порівняно з лабораторною 2

У лабораторній 2 application layer був побудований навколо use case / service-подібних класів. Один клас одночасно визначав вхідні дані, містив orchestration-логіку та часто повертав DTO для API. Це було зручно на старті, але з ростом кількості сценаріїв читання і запису починало змішувати різні відповідальності.

Після рефакторингу в лабораторній 3 проєкт був розділений за принципом CQS:

- `application/commands/...` — команди, які змінюють стан системи;
- `application/queries/...` — запити, які лише читають дані;
- `application/read-models/...` — DTO для читання;
- `infrastructure/prisma/read-repositories/...` — окремі read repositories для query handlers;
- `presentation/http/routes/...` — контролери, що лише перетворюють HTTP-запит у Command або Query і передають його handler-у.

У результаті запис більше проходить через доменну модель (`Item`, `User`) та звичайні репозиторії, а читання може йти напряму через Prisma в read repository, минаючи доменний шар.

## Переваги CQS

### 1. Явне розділення намірів

Тепер із назви класу відразу зрозуміло, що він робить:

- `CreateItemCommandHandler` створює item;
- `BuyItemCommandHandler` виконує покупку;
- `GetItemsQueryHandler` читає список items;
- `GetUserByIdQueryHandler` повертає read model користувача.

Це спрощує навігацію по проєкту й зменшує когнітивне навантаження.

### 2. Команди проходять через домен

Усі операції, що змінюють стан, залишилися в доменній моделі. Наприклад:

- `buyBy()` у `Item` перевіряє, що покупець не купує власний item;
- `withdraw()` і `deposit()` у `User` забезпечують інваріанти балансу;
- `update()` у `Item` перевіряє правило `Type` / `Weapon`.

Це добре, тому що бізнес-правила зосереджені в одному місці, а не розкидані по контролерах і SQL-запитах.

### 3. Запити стали простішими та швидшими

Для читання більше не потрібно обов’язково підіймати доменні сутності. `QueryHandler` може читати оптимізовані поля напряму з БД і повертати `ReadModel`, який уже готовий для API.

Це дає такі плюси:

- менше зайвого мапінгу;
- простіше будувати список або detail view;
- можна повертати саме ту структуру, яка потрібна клієнту.

### 4. Тестування стало природно розділеним

Команди тепер зручно тестувати unit-тестами через fake repositories. Для цього не потрібна БД — достатньо підставити in-memory реалізацію інтерфейсу.

Запити, навпаки, логічно тестувати integration-тестами через HTTP + реальну БД, тому що там важливі:

- правильний SQL / Prisma query;
- селекція полів;
- мапінг у read model;
- фактична відповідь endpoint-а.

## Недоліки CQS

### 1. Збільшилась кількість класів і файлів

Для одного сценарію тепер часто потрібні:

- command/query DTO;
- handler;
- read model;
- інколи окремий read repository.

Для невеликого проєкту це виглядає як додатковий boilerplate.

### 2. Підтримка структури стала суворішою

Потрібно дисципліновано вирішувати, куди покласти нову логіку:

- у домен;
- у command handler;
- у query handler;
- у read repository.

Без чітких правил команда може знову почати змішувати відповідальності.

### 3. Дані після command часто треба перечитувати окремо

Оскільки command не має повертати повні дані сутності, після успішної зміни стану клієнт часто мусить зробити окремий query. Це додає ще один крок взаємодії, але зате зберігає чисте розділення між записом і читанням.

## Чим Command / Query Handler відрізняється від Service, який робить усе сам

Service у стилі лабораторної 2 зазвичай одночасно:

- приймає вхідні дані;
- перевіряє умови;
- читає і змінює стан;
- формує DTO для відповіді.

Через це клас стає «універсальним» і поступово розростається.

`CommandHandler` і `QueryHandler` мають значно вужчу відповідальність:

- `CommandHandler` змінює стан і делегує правила домену;
- `QueryHandler` лише читає й повертає готовий `ReadModel`.

Тобто замість одного «великого» service з’являються маленькі обробники, які простіше змінювати, тестувати й замінювати.

## Як CQS впливає на розширюваність

CQS покращує розширюваність, тому що новий сценарій частіше додається як новий handler, а не як зміна великого існуючого service.

Наприклад, щоб додати нову операцію над items, можна створити окремий command handler, не ризикуючи зламати логіку читання або інші сценарії. Аналогічно, новий спосіб показу списку даних реалізується як окремий query handler або окремий read repository.

Тобто зміни стають локальними:

- нова команда → новий command + handler;
- нове представлення даних → новий query + read model.

Це безпечніше, ніж дописувати ще одну гілку `if` у великий service.

## Чи відрізняється структура даних, яку повертає Query, від доменної моделі

Так, і це принципово важливо.

Доменна модель описує бізнес-сутність та її правила поведінки. Вона не повинна підлаштовуватися під формат HTTP-відповіді.

`ReadModel`, який повертає query, навпаки, орієнтований на потреби клієнта:

- містить лише потрібні поля;
- може мати інші назви полів;
- не містить доменної поведінки;
- може бути оптимізований для списку або detail view.

Це важливо, бо дозволяє незалежно еволюціонувати:

- доменну логіку;
- структуру API-відповідей;
- спосіб читання даних із БД.

## Що стало простіше тестувати порівняно з лабораторною 2, а що складніше

### Простіше

- бізнес-правила команд;
- інваріанти домену;
- сценарії створення, оновлення, архівації, покупки;
- перевірку взаємодії з репозиторіями через fake implementations.

### Складніше

- підтримувати більшу кількість класів;
- синхронізувати command side і query side;
- писати більше «каркасного» коду для DTO, handler-ів і read repositories.

## Висновок

CQS зробив структуру application layer більш явною та масштабованою. Команди стали відповідати лише за зміну стану, а запити — лише за читання. Це покращило тестованість, ізолювало бізнес-правила в домені та дало змогу повертати окремі read models для API.

Основний мінус підходу — більша кількість файлів і дещо вищий поріг організаційної складності. Проте для проєкту, який розвивається, така структура є вигіднішою за універсальні service-класи, що намагаються робити все одразу.
