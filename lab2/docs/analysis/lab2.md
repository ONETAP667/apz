# Аналіз лабораторної роботи 2

## Що змінилося порівняно з лабораторною 1

У першій лабораторній проєкт мав просту layered-структуру: routes, services, validation, Prisma client та utils. Контролери були відносно тонкими, але сервіси напряму залежали від Prisma. Бізнес-логіка покупки предмета, перевірки власника, балансу, архівності buyer/seller та транзакція знаходилися в `ItemsService`. Валідація простих правил була винесена в окремі функції, але ці функції кидали загальну application/http-помилку.

У другій лабораторній проєкт перероблено на чотири шари:

- `presentation` — HTTP routes, middleware, error mapping;
- `application` — use cases, які оркеструють домен і репозиторії;
- `domain` — доменні моделі, value objects, domain errors, інтерфейси репозиторіїв;
- `infrastructure` — Prisma repositories, ORM-мапери, bcrypt, JWT.

Тепер доменна модель не є ORM-моделлю. Наприклад, `User` та `Item` — це окремі класи з поведінкою та інваріантами, а Prisma-моделі використовуються тільки в infrastructure layer.

## Які переваги дало розділення на шари

Перша перевага — бізнес-логіка стала незалежною від БД та HTTP. Доменні класи `User`, `Item`, `Email`, `Money` не імпортують Express, Prisma, JWT або bcrypt. Це відповідає правилу залежностей: домен не залежить від зовнішніх шарів.

Друга перевага — з'явилися чіткі use cases. Наприклад, покупка предмета реалізована через `BuyItemUseCase`, який отримує `ItemRepository` та `UserRepository` через інтерфейси. Use case не знає, чи дані зберігаються у PostgreSQL, MongoDB або in-memory сховищі.

Третя перевага — тести стали простішими на рівні домену. Доменні тести запускаються без БД і без HTTP-сервера. Для складних інваріантів, які потребують перевірки сховища, можна використовувати fake/in-memory реалізації репозиторіїв.

Четверта перевага — помилки домену більше не залежать від HTTP. Домен кидає `ValidationError`, `ConflictError`, `NotFoundError`, `AuthenticationError`, а presentation layer мапить ці помилки у HTTP-статуси.

## Які недоліки та ускладнення з'явилися

Головний недолік — стало більше коду. Замість одного service-файлу тепер є domain entity, value objects, factory, repository interface, use case, mapper і repository implementation.

Другий недолік — потрібен мапінг між ORM-моделями та доменними моделями. Наприклад, Prisma використовує поля `ID`, `Username`, `Balance`, а доменна модель працює з `id`, `username`, `balance`. Це додає код, але дозволяє не змішувати persistence model і domain model.

Третій недолік — складніша навігація. Щоб зрозуміти повний шлях запиту, треба пройти route → use case → domain model/factory → repository interface → Prisma repository → mapper.

## Наскільки простіше змінити БД або фреймворк

Змінити БД стало простіше, тому що Prisma ізольована в `infrastructure/prisma`. Якщо потрібно перейти з PostgreSQL/Prisma на MongoDB, основні зміни будуть у реалізаціях `UserRepository`, `ItemRepository` та mapper-класах. Доменні моделі, use cases і HTTP routes можна залишити майже без змін.

Змінити HTTP-фреймворк також стало простіше. Express використовується тільки в presentation layer. Якщо замінити Express на Fastify або NestJS, application і domain layers не мають змінюватися.

## Чому обрано Rich Domain Model

Обрано Rich Domain Model, тому що система має поведінку, яку краще тримати всередині доменних об'єктів:

- `User.withdraw()` перевіряє активність користувача та достатність балансу;
- `User.deposit()` змінює баланс продавця;
- `Item.create()` перевіряє правило `Type / Weapon`;
- `Item.buyBy()` перевіряє, що предмет активний, покупець і продавець активні, покупець не є власником предмета, після чого змінює власника.

Anemic model теж міг би працювати, але тоді вся логіка залишилася б у use cases або service-класах. Для цієї лабораторної Rich Domain Model краще демонструє відділення бізнес-логіки від інфраструктури.

## Висновок

Після рефакторингу проєкт став ближчим до Clean/Hexagonal Architecture. Домен більше не залежить від ORM та HTTP, бізнес-правила можна тестувати без інфраструктури, а заміна БД або фреймворку стала локалізованою задачею. Ціна цього рішення — більша кількість файлів, необхідність mapper-класів і складніша структура проєкту.
