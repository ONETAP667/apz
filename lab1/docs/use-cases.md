# 1. Реєстрація користувача

**Актор:** гість системи

**Передумови:**
- користувач ще не зареєстрований у системі
- email не використовується іншим користувачем
- заповнені обов’язкові поля: Username, email, password

**Основний сценарій:**
1. Гість надсилає запит на реєстрацію.
2. Система перевіряє наявність обов’язкових полів.
3. Система перевіряє формат email.
4. Система перевіряє, чи не існує користувач із таким email.
5. Система хешує пароль.
6. Система створює нового користувача.
7. Система заповнює службові поля значеннями за замовчуванням:
   - Balance = 0
   - DateJoined = поточна дата і час
   - Archive = false
   - Version = 0
8. Система повертає дані користувача та токен доступу.

**Альтернативні сценарії / помилки:**
- якщо Username, email або password не передані — `400 Bad Request`
- якщо email має невалідний формат — `400 Bad Request`
- якщо користувач із таким email уже існує — `409 Conflict`

3. Система перевіряє хеш пароля.
4. Система формує токен доступу.
5. Система повертає токен та дані користувача.

**Альтернативні сценарії / помилки:**

# Купівля предмета

**Актор:** авторизований користувач

## Передумови
- користувач авторизований (має валідний JWT)
- предмет існує
- предмет не знаходиться в архіві
- користувач не є власником предмета
- у користувача достатньо коштів для покупки

## Основний сценарій
1. Користувач надсилає запит `POST /items/:id/buy`.
2. Система перевіряє JWT-токен.
3. Система отримує предмет за ID.
4. Система перевіряє, що предмет існує.
5. Система перевіряє, що користувач не є власником предмета.
6. Система перевіряє баланс користувача.
7. Система виконує транзакцію:
   - списує кошти з покупця
   - нараховує кошти продавцю
   - змінює власника предмета
8. Система повертає результат операції.

## Альтернативні сценарії / помилки
- `401 Unauthorized` — користувач не авторизований
- `404 Not Found` — предмет не знайдено
- `400 Bad Request`:
  - користувач намагається купити власний предмет
  - недостатньо коштів
  - предмет знаходиться в архіві

# Основні інваріанти

## Для Users
- Username є обов’язковим і не може бути порожнім
- email є обов’язковим
- email повинен мати валідний формат
- email повинен бути унікальним
- password є обов’язковим при реєстрації
- password_hash зберігається тільки у хешованому вигляді
- Balance не може бути від’ємним
- password_hash ніколи не повертається у відповіді API

## Для Items
- OwnerID є обов’язковим і має посилатися на існуючого користувача
- Price не може бути від’ємним
- якщо Type = 1, то Weapon є обов’язковим
- якщо Type != 1, то Weapon повинен бути NULL
- предмет повинен існувати перед оновленням або видаленням
- Archive використовується для логічного видалення, якщо така поведінка реалізована в API
- користувач не може купити власний предмет
- баланс користувача не може стати від’ємним
- операція покупки виконується в межах транзакції
- після покупки змінюється власник предмета

## Додаткові бізнес-правила
- лише користувач Admin може створювати нові предмети
- звичайні користувачі не можуть створювати предмети
- користувач не може купити власний предмет
- баланс користувача не може бути від’ємним після покупки
