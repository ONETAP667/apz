# ADR 0001: Вибір Rich Domain Model

## Статус
Прийнято.

## Контекст
У першій лабораторній бізнес-правила були розміщені переважно у сервісах та окремих validation-функціях. Для другої лабораторної потрібно відокремити домен від ORM, HTTP та інфраструктури, а також реалізувати доменні моделі, фабрики, репозиторії через інтерфейси та доменні помилки.

## Рішення
Обрано підхід **Rich Domain Model**.

Причина: у системі є не лише CRUD, а й інваріанти та поведінка:
- email має бути валідним;
- username не може бути порожнім;
- balance не може бути від'ємним;
- price не може бути від'ємною;
- якщо `Type = 1`, то `Weapon` обов'язковий;
- якщо `Type != 1`, то `Weapon` має бути `null`;
- користувач не може купити власний предмет;
- покупка змінює власника предмета та баланси покупця і продавця.

Ці правила належать предметній області, тому вони розміщені у доменних об'єктах `User`, `Item`, value objects `Email`, `Money` та фабриках `UserFactory`, `ItemFactory`.

## Наслідки
Переваги:
- домен не залежить від Prisma, Express або HTTP;
- бізнес-логіку можна тестувати без БД;
- легше замінити БД або HTTP-фреймворк;
- правила зібрані ближче до сутностей, яких вони стосуються.

Недоліки:
- більше файлів і коду;
- потрібні мапери між ORM-моделями та доменними моделями;
- навігація складніша, ніж у простій CRUD-структурі.
